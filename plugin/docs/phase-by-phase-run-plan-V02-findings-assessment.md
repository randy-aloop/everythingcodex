# Phase-by-Phase Run Plan V02 Findings Assessment

Report Date: 2026-06-23
Target: `docs/phase-by-phase-run-plan.md`
Target Version: V02
Action: Review only; no patch applied to the target run plan.

## Executive Assessment

The pasted findings are largely correct. The two leading issues are real documentation/control-flow errors:

1. Phase 0 tells the controller to write `.qc/issue-register.jsonl`, `phase-record.md`, and the deviation log before `.qc/` and the phase run files are created.
2. The body phase numbering and the One-Page Run Order diverge from Ponytail onward because the run order elevates diff persistence into a separate numbered step while the body keeps it inside Phase 3.

Most smaller findings are also valid. Two are partial rather than absolute: `record_decision.py` does need a defined decision id source, but timestamp may reasonably be generated by the script; and in-progress validation behavior is partly defined by current code, but the doc does not state the exit-code contract clearly enough.

## Finding Assessment

### 1. Phase 0 Writes To Files That Do Not Exist Yet

Verdict: Correct.

Evidence in the doc:

- Phase 0 action 6 says to record assumptions in the phase record or deviation log.
- Phase 0 says to record the pre-build plan check in `phase-record.md` and open blockers in `.qc/issue-register.jsonl`.
- Phase 1 creates or verifies `.qc/`.
- Phase 2 creates the phase run folder and `phase-record.md`.
- The One-Page Run Order runs pre-build plan check before `init_qc.py` and `start_phase.py`.

Assessment:

As written, Phase 0 requires durable writes before the storage targets exist. This is a real sequencing bug. The fix should either move `init_qc.py` before the pre-build write step, or make Phase 0 collect findings in memory and persist them only after Phase 1 and Phase 2 create the records.

### 2. Body Phase Numbers And One-Page Run Order Are Desynchronized

Verdict: Correct.

Evidence in the doc:

- Body Phase 3 includes builder output and diff evidence.
- Body Phase 4 is Ponytail.
- Body Phase 7 is strict gate.
- Body Phase 12 is handoff.
- One-Page Run Order makes `persist changed-files/diff evidence` step 4, Ponytail step 5, strict gate step 8, and handoff step 13.

Assessment:

This is a real cross-reference hazard. A reader following body phases and checklist steps could refer to different operations using the same number. The clean fix is to either make the One-Page Run Order use the same phase numbers as the body, or renumber the body so diff persistence becomes its own phase everywhere.

### 3. `not_applicable` Falls Through The Strict-Gate Grammar

Verdict: Correct as a documentation gap.

Evidence in the doc:

- Release-agent gate expectation allows `not_applicable` when `release_required=false`.
- Phase 5E repeats that `Verdict: not_applicable` is allowed only when `release_required=false`.
- Strict gate blocks missing, duplicated, unknown, `pending`, `revise`, or `block` role verdicts.

Assessment:

The doc recognizes `not_applicable` in release-specific text, but the strict-gate section does not explicitly carve out the conditional release case. The result is ambiguous: a strict-gate implementer could treat `not_applicable` as an unknown or non-pass required verdict. The strict-gate rule should state that `not_applicable` is accepted only for `release-gate.md` when `release_required=false` and a rationale is present.

### 4. `required_evidence` Is Incomplete

Verdict: Correct.

Evidence in the doc:

- The Phase 2 phase-board example includes `phase-record.md`, `builder-notes.md`, diff artifacts, Ponytail events, test JSONL, reviewer, compliance, and seam audit.
- It omits `test-report.md`.
- It omits `gate-summary.md`.
- It omits `release-gate.md`, including the conditional case where `release_required=true`.

Assessment:

The finding is correct. The example required-evidence list is not complete enough to drive deterministic validation. It should include human-readable and machine-readable evidence, and it should define conditional evidence such as `release-gate.md` when release is required.

### 5. "Expected State" Trees Show Files The Tools Do Not Create

Verdict: Correct.

Evidence in the doc:

- Phase 1 "Expected state" includes `lessons-learned.jsonl`.
- Immediately after, the doc says `init_qc.py` does not yet seed `lessons-learned.jsonl`.
- Phase 2 "Expected state" includes `changed-files.json`, `implementation-diff.patch`, and `gate-summary.md`.
- Immediately after, the doc says `start_phase.py` does not create those files.

Assessment:

The doc is honest about the helper gaps, but the label "Expected state" is misleading. It reads like tool output. A better label would be "Target layout" or "Target state after helper plus manual controller records." Manual items should be marked directly in the tree.

### 6. `lessons-learned.jsonl` Is Write-Only

Verdict: Correct.

Evidence in the doc:

- Phase 11 says to append reusable process lessons to `.qc/lessons-learned.jsonl`.
- No earlier phase, especially Phase 0 intake, says to read or apply prior lessons.
- No role contract in this run plan consumes lessons.

Assessment:

Correct. The doc captures lessons but does not feed them forward. That makes the self-improvement loop incomplete. Phase 0 should ingest open/applicable lessons, or the controller should explicitly state that lessons are archival only until a future implementation consumes them.

### 7. `record_decision.py` Has No `--decision-id` Or Timestamp Contract

Verdict: Partially correct, with the decision-id issue confirmed.

Evidence in the doc:

- Required decision-log fields include `decision_id`.
- Phase 9 says `accepted_with_risk` must reference a human decision id in deviation or issue records and `phase-board.json`.
- The target `record_decision.py` command does not include `--decision-id`.
- The doc does not say the helper returns or prints a generated decision id.
- The command also has no timestamp argument.

Assessment:

The missing decision id source is a real contract gap. The timestamp point is weaker because existing helper patterns often generate timestamps internally; a CLI flag is not strictly required. But the doc should state whether `record_decision.py` generates `timestamp` and `decision_id`, accepts `--decision-id`, or prints the generated id for cross-reference.

### 8. In-Progress Validation Vs Exit Codes Is Underspecified

Verdict: Partially correct.

Evidence in the doc:

- Phase 6 says warnings are allowed while role reports are pending.
- The exit-code section says the current validator returns `0` when no errors are present and `1` for any validation error.
- Current `validate_phase_record.py` behavior confirms pending role reports become warnings when `--strict-gate` is absent, while missing required files, missing Ponytail event, failed tests, blocker deviations, release errors, and safety findings can still be errors.

Assessment:

The pasted finding is right that the doc does not clearly state the non-strict exit-code rule. However, current implementation does distinguish warning-only pending reports from errors: warnings alone do not make the validator return `1`. The fix is documentation clarity: non-strict validation should be defined as exit `0` for warnings-only and exit `1` for actual errors under the current implementation, with target richer exit codes reserved for strict gate.

### 9. `release_not_applicable_rationale` Has No Value Contract When `release_required=true`

Verdict: Correct.

Evidence in the doc:

- Phase-board examples all use `release_required=false`.
- The placeholder says `release_not_applicable_rationale` is required when `release_required=false`.
- No example or rule says whether the field should be empty, omitted, or set to a runtime/release summary when `release_required=true`.

Assessment:

Correct. The schema needs a runtime-phase value rule. Recommended contract: keep the field present as an empty string when `release_required=true`, or rename/split the field so the runtime case has `release_gate_evidence` or `release_gate_status` instead.

## Additional Observation

The file is versioned as document V02, but line-level wording still says "V01 is local-only." That may be intentional if V02 is a documentation version describing the V01 runtime. It is not necessarily an error, but it is likely to confuse readers unless the document explicitly distinguishes "Docs V02" from "Runtime V01."

## Recommended Fix Order

1. Fix Phase 0 persistence sequencing.
2. Align body phase numbers and One-Page Run Order.
3. Clarify strict-gate handling for release `not_applicable`.
4. Complete the `required_evidence` examples and conditional release evidence.
5. Rename misleading "Expected state" labels or mark manual files in the tree.
6. Add lessons-read behavior to Phase 0 or mark lessons archival.
7. Define `record_decision.py` id generation/return contract.
8. Clarify non-strict validator warning/error exit behavior.
9. Define `release_not_applicable_rationale` when `release_required=true`.
