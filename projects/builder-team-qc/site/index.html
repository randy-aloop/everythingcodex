Ponytail is not a test-agent helper. It is a phase-scope gate that feeds reviewer, compliance, and the final controller decision.

What is a builder-team multi-agent system?

A builder-team system is a structured way for Codex to run a build like a small engineering team. The point is not to create chaos with many agents talking at once. The point is to assign stable responsibilities, write evidence to disk, and make the phase gate inspectable after the chat is over.

In builder-team-qc, the phase is the unit of control. Each role sees the current phase through its own lens, writes a record, and hands control back to the phase controller.

Local control

Codex stays in the loop. No hidden server, remote A2A surface, public tunnel, or automatic global install is required for V01.

Role clarity

Builder, reviewer, test, compliance, integration, release, and Ponytail roles each produce evidence instead of all blending into one chat narrative.

Durable proof

The .qc/ folder records phase runs, tests, deviations, decisions, seam audits, Ponytail checks, and gate outcomes.

Codex agent types for this system

The Google ADK article groups agents into LLM, workflow, and custom logic. Builder Team QC mirrors that mental model, but implements it as local Codex skills and scripts.

Workflow agent phase-controller Controls sequence, fan-out, revise loop, and final gate decision.

The foundational concept: hierarchy

Builder Team QC has one visible controller and a stable set of role contracts. The controller may apply roles one at a time or group independent checks, but it should not let any role advance the build without the gate.

Orchestrating work with workflow patterns

ADK uses Sequential, Parallel, and Loop agents. Builder Team QC uses the same orchestration ideas in a local file-backed way. In V01, fan-out is sequential role execution with a join point, not true concurrent runtime execution.

Sequential

plan → start phase → build → Ponytail gate → validate

Parallel-style checks

Logical fan-out only: Codex applies these checks one at a time unless a future runtime adds real concurrency.

test review compliance seam audit release gate → join

Loop

The revise loop is capped at three repeated failed attempts before the controller blocks and asks for a human decision.

strict gate fails → fix smallest item → re-test → validate again
The hard stop is the deterministic validator: validate_phase_record.py --strict-gate must exit cleanly for evidence completion. Model-authored role reports are useful review evidence, but executable checks and validator exit codes carry more weight.

How agents communicate: .qc shared state

ADK has shared session state. Builder Team QC uses a project-local .qc/ folder. It is a durable whiteboard for phase state, role outputs, tests, deviations, accepted risks, and final gate decisions.

phase-board.jsonCurrent phase, status, next phase, required evidence, and latest gate.
phase-runs/{phase-id}/Phase record and role reports for builder, reviewer, test, compliance, seam, and release.
test-results/{phase-id}.jsonlMachine-readable test evidence with command, status, exit code, and notes.
ponytail-events.jsonlPonytail mode, checks, and minimal-code verdict.
deviation-log.jsonlScope changes, blockers, unresolved risk, and accepted-risk metadata.
decision-log.jsonlHuman decisions, approvals, accepted-risk bypasses, and follow-up commitments.

accepted_with_risk is a gate bypass, not a controller convenience. It should require an explicit human decision recorded in decision-log.jsonl; the controller must not self-approve incomplete evidence.

Ponytail is a gate, not a test helper

Ponytail is the project code name for the minimal-code discipline checklist used by Builder Team QC. In V01, it is an instruction/checklist adapter, not a trusted external runtime or autonomous hook system.

Credit: the Ponytail discipline in this project credits DietrichGebert/ponytail, the MIT-licensed Ponytail project by Dietrich Gebert. Builder Team QC does not vendor or run upstream Ponytail by default; it uses a local checklist unless upstream hooks are explicitly reviewed, enabled, and recorded.

The Ponytail adapter checks whether the builder output obeys the minimal-code discipline: YAGNI, standard library first, existing project tools before new dependencies, and no unnecessary abstraction. If it passes, the evidence checks fan out. If it revises or blocks, the controller should stop or loop back before spending time on deeper validation.

QC

Gate rule

The latest Ponytail event for the phase must be present and must be pass. Reviewer and compliance roles read that event as evidence for final gate validation.

Agent-as-a-tool vs sub-agent

A role skill is a permanent member of the builder-team hierarchy. A script is an explicit tool called for a narrow action. Keeping this boundary clear prevents helper code from taking over the phase.

Feature Role skill Script tool
Example reviewer-agent record_test_result.py
Who stays in control? phase-controller phase-controller
What it does Applies a reasoning contract and writes a report. Performs a precise record or validation action.
Can it advance the phase? No. It produces evidence. No. It records or checks evidence.

How to run it

Invoke the process from Codex with the target root, build plan, and current phase. Codex reads the plugin docs, initializes .qc, starts the phase record, applies roles, records evidence, and validates the strict gate.

Use builder-team-qc for this build.
Project path: https://github.com/randy-aloop/everythingcodex/tree/main/projects/builder-team-qc
Build plan: path\to\build-plan.md
Current phase: phase-001 - Scaffold local plugin
Run the phase controller with Ponytail minimal-code checks,
tests, seam audit, compliance review, and strict gate validation.

For command-level operation, the key scripts are init_qc.py, start_phase.py, record_ponytail_check.py, record_test_result.py, validate_phase_record.py, and summarize_phase.py.
